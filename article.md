> 宁波睿威工程技术有限公司技术团队

---

做过工程监测的人都清楚一个现实：外业采集数据可能只花半天，但回到办公室后的闭合差校核、平差计算、趋势分析、日报撰写、规范审查，往往要消耗两到三倍的时间。更棘手的是，这些内业环节高度依赖个人经验——一个刚入行的测量员可能会把四等水准的限差公式记错，一份赶工出来的监测周报可能漏引了最新版的 GB 50911 条文。

我们团队长期从事城市轨道交通保护区监测、深基坑支护监测和工程控制测量，深知这些痛点不是靠"上一套软件"就能解决的。市面上的测绘软件解决的是计算问题，但**从原始数据到甲方交付物之间的判断、协调和审查**，始终需要人来完成。

2025年初，我们开始了一个内部实验：能不能用大语言模型（LLM）搭建一个"数字项目组"，让它像真实的工程团队一样，分角色、按流程、有质检地完成内业全链路？

经过近一年的开发迭代，这个项目以 **RAILWISE-CLI** 的形态落地——一个运行在终端里的多智能体 AI 系统，内置 7 个专业智能体和 19 个本地计算工具。更重要的是，它拥有**跨会话记忆能力**——系统会自动记住你的项目结构、计算习惯和工作偏好，**用得越多越顺手，就像一个不断成长的工程搭档**。本文将详细介绍它的架构设计和核心机制，希望能给同行一些参考。

## 一、为什么是"多智能体"而不是"一个AI助手"？

直接用 ChatGPT 或 DeepSeek 聊天处理测绘数据，很多人试过，体验通常是这样的：它能听懂你的问题，甚至能背出规范条文，但让它算一个水准网的最小二乘平差——结果大概率是错的。大模型的本质是语言模型，它擅长的是理解、推理和表达，而不是严密的数值计算。

更根本的问题在于：**一个真实的监测项目从来不是一个人完成的。**

以一份监测日报为例，完整的工作流至少涉及：外业数据质检→内业平差计算→趋势分析→报告撰写→规范审查。在现实中，这分别由质检员、数据处理员、报告编写人员和总工来完成。每个角色有不同的专业背景、不同的关注点、不同的审查标准。

单一的 AI 助手试图一个人扮演所有角色，结果就是——哪个角色都演不好。它在计算时忘了审查规范，在写报告时又忘了它刚才算出来的数据是否经过质检。

**RAILWISE-CLI 的核心设计理念，就是把这种真实的项目组分工搬进了 AI 系统。**

我们为系统配置了7个独立的智能体（Agent），每个智能体有明确的角色定位、专属的系统提示词（System Prompt）和严格的执行边界。它们之间通过一个调度中枢进行协调，按照预定义的 SOP（标准作业程序）流转工作。

【配图2：多智能体协作与真实项目组的类比对照图——左侧为传统项目组（项目经理、方案工程师、数据处理员、报告编写、外业质检、内业审核、商务专员），右侧为对应的7个AI智能体。用虚线标注一一对应关系。】

## 二、系统架构：7个专家 + 19把算盘

### 2.1 智能体团队

| 智能体 | 角色定位 | 核心职责 |
|--------|---------|---------|
| `chief_manager` | 项目总工 | 任务拆解（WBS）、智能体调度、成果汇总。**禁止自己编造任何技术数据** |
| `solution_architect` | 方案设计师 | 监测方案编制、测点布设、仪器选型、技术路线规划 |
| `data_analyst` | 数据分析师 | 粗差剔除、最小二乘严密平差、变形趋势拟合、分级预警研判 |
| `qa_inspector` | 外业质检员 | 第一道防线：原始数据完整性审查、闭合差限差核算。超限零容忍 |
| `qa_reviewer` | 内业审核员 | 第二道防线：规范合规性终审（GB 50911/GB 50026/JGJ 8），拥有最高否决权 |
| `technical_writer` | 技术文档员 | 将计算结果转化为甲方级交付报告（日报/周报/月报） |
| `commercial_specialist` | 商务专员 | 招投标响应、合同风险审核、计量催款 |

这些智能体不是简单的"换了个名字的 ChatGPT"。每个智能体的提示词都经过精心设计，融入了大量行业实操经验。例如，`qa_inspector` 的提示词中明确规定：水准测量必须检查"后-前-前-后"观测顺序、前后视距差不得超过规范限值（二等≤1m，三等≤2m，四等≤5m）；`qa_reviewer` 被要求在审查时强制核对 GB 50911、GB 50026 等六部核心规范的具体条文。

### 2.2 专业计算工具

智能体负责理解、判断和表达，但具体的数值计算全部交给本地的 TypeScript 工具函数执行。我们为系统开发了 **19 个专业计算工具**，覆盖测绘内业的主要计算场景：

**基础测量计算**
- `survey_calculator`：水准/导线闭合差校核、最小二乘严密平差、预警等级判定
- `coord_transform`：高斯-克吕格正反算、七参数布尔莎坐标系转换
- `angle_convert`：度分秒/十进制度/弧度/密位/百分度互转
- `distance_calculator`：气象改正→斜距化平→投影改正的全流程归算
- `control_network`：平面控制网间接平差，输出平差坐标、精度评定与误差椭圆

**施工与放样**
- `pile_stakeout`：极坐标放样、里程偏距计算、批量放样点生成
- `shield_guidance`：盾构机姿态解算——根据头尾实测坐标计算中心、方位角、俯仰角、滚转角，与设计轴线对比偏差
- `cpiii_adjustment`：CPIII 自由测站后方交会，多已知点解算测站坐标与方位定向

**监测分析**
- `deformation_rate`：变形速率计算、线性回归趋势预测、多测点对比
- `cross_section`：隧道收敛量计算、断面超欠挖分析、建筑限界检查
- `monitoring_csv`：自动化监测 CSV 海量数据清洗与统计
- `axial_force`：基坑钢/混凝土支撑轴力计算，根据频率或应力值换算实际轴力并分级预警
- `inclinometer`：测斜仪深层水平位移剖面计算，生成位移-深度曲线
- `water_level`：地下水位变化量与变化速率分析，判断降水效果和回灌影响

**数据处理与输出**
- `format_parser`：徕卡 GSI-8/GSI-16 及 DAT 格式解析
- `chart_generator`：SVG 趋势折线图生成
- `report_export`：Markdown 转 DOCX 报告、EXCEL 报表导出
- `excel_export`：监测数据导出为 .xlsx 文件，支持多 Sheet、表头冻结、自动筛选、数值格式化与预警标识
- `standard_query`：工程规范条文智能查询

## 三、核心机制深度解读

### 3.1 项目总工的调度逻辑

`chief_manager` 是整个系统的大脑。当用户下达任务指令后，它首先输出一份 **任务拆解与执行计划（WBS）**，明确列出将调用哪些智能体、执行顺序和预期产出，然后按计划逐一调度。

关键约束是：**chief_manager 被严格禁止自己生成任何具体的技术数据或计算结果。** 如果用户问"帮我算一下这条水准路线的闭合差"，它不会自己去算，而是将任务分派给 `data_analyst`，后者再调用 `survey_calculator` 工具完成精确计算。

这种"总工只调度、不动手"的设计，确保了每一步计算都经过专业的工具函数执行，每一份报告都经过专职审核，避免了单一 AI 对话中常见的"一边算一边忘"的问题。

### 3.2 双重质检机制

这是我们认为最有实际价值的设计。系统中设置了两道独立的质检关卡：

**第一道：外业数据首检（`qa_inspector`）**

所有原始观测数据在进入内业计算之前，必须先经过 `qa_inspector` 的审查。它会检查：
- 数据包是否完整（仪器参数、测站信息、原始观测值是否齐全）
- 仪器是否在有效检定期内
- 观测步骤是否合规（水准的 BFFB 顺序、前后视距差等）
- 闭合差是否在限差范围内

审查不通过，数据直接打回，流程暂停，等待外业人员补充或返工。

**第二道：内业成果终审（`qa_reviewer`）**

所有准备交付的技术报告在输出前，必须经过 `qa_reviewer` 的规范审查。它会逐条对照 GB 50911、GB 50026、JGJ 8 等现行标准，检查监测项目是否缺失、报警值设定是否合理、术语是否准确、数值是否前后一致。

`qa_reviewer` 拥有系统中的最高否决权——如果发现涉及工程安全的问题（如深基坑缺少周边地表沉降监测项目），即使 `chief_manager` 已经将报告标记为"完成"，`qa_reviewer` 也会直接驳回并要求重写。

【配图4：双重质检机制流程图——外业原始数据 →【关卡1】qa_inspector 首检（通过/驳回返工）→ data_analyst 平差计算 → technical_writer 报告编制 →【关卡2】qa_reviewer 终审（通过/驳回重写）→ 最终交付。两个关卡用红色醒目标注。】

### 3.3 "让代码算、让 AI 判"

大语言模型有一个被反复验证的弱点：**数学幻觉**。它能完美复述闭合差限差公式 $F_h = \pm 20\sqrt{L}$，但让它实际算 $20 \times \sqrt{2.5}$ 的结果，有不低的概率会算错。

我们的解决方案是彻底分离"理解"和"计算"两个环节：

- **AI 负责理解语义**：从用户输入中准确提取参数
- **本地代码负责精确计算**：调用 TypeScript 工具函数执行数学运算
- **AI 负责解读结果**：用专业的工程语言撰写结论

以一个真实场景为例：

> 外业队长输入：*"今天跑完了 XX 站的沉降水准网，路线全长 2.5 公里，四等水准，实测闭合差 35 毫米。"*

系统的内部流转是这样的：

1. `qa_inspector` 理解语义，提取参数：`measuredError=35mm`，`routeLengthKm=2.5`，`order=4th`
2. `qa_inspector` 调用 `leveling_closure` 工具，参数传入本地函数
3. 工具函数在毫秒内精确计算：$20 \times \sqrt{2.5} = 31.623\text{mm}$，判定 $|35| > 31.623$，返回 `is_passed: false`
4. `qa_inspector` 基于工具返回结果输出审查意见：

> **【拒绝接收】** 本次四等水准测量路线长 2.5km，规范允许限差为 ±31.62mm（公式：±20√L = ±20×√2.5）。实测闭合差 35mm，超出限差 3.38mm。请立即检查测段数据并安排返工重测！

整个过程中，AI 没有做任何数学运算——所有涉及数字的部分都由经过验证的本地代码完成。这种机制保证了工程级的计算精度。

【配图5：工具调用机制示意图——分为5个步骤：① 用户自然语言输入 → ② AI理解语义并提取参数 → ③ 调用本地TypeScript工具函数 → ④ 工具返回JSON结构化结果 → ⑤ AI用专业术语解读并输出。底部标注"AI负责理解和表达，代码负责计算"。】

### 3.4 为什么是"工具"而不是"技能"？——两种扩展机制的本质区别

熟悉 AI Agent 系统的读者可能会问：RAILWISE-CLI 的 19 个专业计算模块为什么全部做成了 **Tool（工具函数）**，而不是用更时髦的 **Skill（技能提示词）** 来实现？

这涉及到系统中两种完全不同层次的扩展机制。理解它们的区别，对于理解整个架构的设计逻辑至关重要。

**Tool = 手**

工具是一段真实执行的 TypeScript 代码。AI 提取参数，传给函数，函数在本地毫秒级完成计算，返回结构化的 JSON 结果。整个过程中 AI 不参与任何数学运算——它只负责"听懂需求"和"解读结果"。

以 `leveling_closure`（水准闭合差校核）为例，它的核心就是一个确定性的数学函数：

```typescript
// 接收三个参数：实测闭合差、路线长度、测量等级
// 返回：限差值、是否合格、超限量等结构化结果
const limit = k * Math.sqrt(routeLengthKm)
const pass = Math.abs(measuredError) <= limit
```

这段代码无论执行一万次结果都一样。$20 \times \sqrt{2.5}$ 永远等于 $31.623$，不会有任何"幻觉"。

**Skill = 操作手册**

技能则是一个 Markdown 文档（`SKILL.md`），被加载后注入到 AI 的对话上下文中。它不执行任何计算，只提供指令和工作流描述——告诉 AI "遇到这种情况应该怎么做"。

比如可以写一个"外业数据质检"技能，里面描述：第一步检查数据完整性，第二步调用 `survey_calculator` 校核闭合差，第三步生成质检报告。但技能本身不做任何校核——它只是一份行为指南。

**为什么测绘计算必须用 Tool？**

| 维度 | Tool（代码执行） | Skill（提示词注入） |
|------|-----------------|-------------------|
| 计算精度 | 确定性，零误差 | LLM 有数学幻觉风险 |
| 执行方式 | 本地代码运行，毫秒级 | 文本塞进上下文，消耗 token |
| 适合场景 | 数值计算、数据解析、格式转换 | 工作流编排、SOP 指导、知识注入 |
| 工程安全 | 可审计、可测试、可复现 | 依赖 AI 理解准确度 |

**简单说：闭合差超不超限，这件事不能让 AI "想"，必须让代码"算"。**

工程测绘的计算容错空间极小——闭合差算错一个小数点就是返工重测，坐标转换差一个参数就是打桩偏位。这类场景下，工具函数的确定性执行是不可妥协的底线。

当然，两者并不矛盾。在 RAILWISE-CLI 中，它们是互补关系：**Skill 负责编排"做什么、按什么顺序做"，Tool 负责执行"具体怎么算"。** 就像一个项目经理拿着 SOP 手册（Skill）指挥，每个计算环节都由专人（Tool）精确完成——这也正是真实工程项目组的运作方式。

【配图：Tool 与 Skill 协作关系图——左侧为 Skill（SKILL.md 文档），标注"工作流指令"，箭头指向 AI 大脑。右侧为 Tool（.ts 代码），标注"精确执行"，箭头从 AI 大脑指出。中间的 AI 大脑连接两端，上方标注"理解+判断+表达"。底部对比标语："Skill 告诉 AI 该做什么，Tool 保证每一步算对。"】

### 3.5 跨会话记忆：越用越懂你

传统的 AI 对话工具有一个令人头疼的问题：**每次新开对话，它什么都不记得了。** 昨天你花了半小时告诉它你们项目用的是四等水准、CGCS2000 坐标系、报警阈值按 GB 50911 取值——今天开个新对话，又要从头说一遍。

我们管这叫"金鱼问题"——记忆只有一个会话那么长。

RAILWISE-CLI 通过内置的**跨会话记忆系统**彻底解决了这个问题。它的工作机制是这样的：

1. **自动提取**：每次会话结束或触发上下文压缩时，系统自动从对话摘要中提取有价值的知识——你提到的项目参数、使用的坐标系、报告的格式偏好、踩过的坑
2. **智能去重**：新提取的记忆会和已有记忆进行相似度比对（Jaccard 算法，阈值 0.7），重复内容不会冗余存储，而是提升原有记忆的置信度
3. **按需注入**：下次开启新会话时，系统自动将最相关的记忆注入上下文，AI 从第一句话就了解你的项目背景
4. **自然淘汰**：高频被验证的记忆自动提升优先级，长期未被使用的记忆逐渐淡出

实际效果是这样的：

> **第一周**：你告诉系统"我们这个项目是地铁 3 号线保护区监测，四等水准，CGCS2000 坐标系，沉降报警值 ±30mm"
> 
> **第二周**：你直接说"分析一下今天的沉降数据"——它已经知道是哪个项目、什么等级、什么报警值了
> 
> **一个月后**：你只需要说"出日报"——项目参数、报告模板、甲方偏好的表述风格、上次被审核打回的原因……它全都记得

**用得越多，交互越精简，效率越高。** 这不是"每次从零开始"的一次性工具，而是一个在持续积累中变得越来越好用的工程伙伴。

### 3.6 飞书集成：打通办公知识库与 AI 工程助手

```bash
railwise feishu
```

按引导输入飞书应用凭证（App ID + App Secret），选择需要的功能模块，系统就会自动配置好飞书 MCP 连接。之后，AI 智能体就能直接：

- **读取飞书云文档** — 将技术方案、交底记录、操作规程等企业知识自动纳入 AI 上下文
- **检索知识库** — 搜索企业公开知识库中的历史项目资料、规范解读、经验总结
- **创建多维表格** — 将监测数据自动写入飞书多维表格，实现数据的在线协作与共享
- **发送消息通知** — 当监测数据触发预警时，自动将告警信息推送到指定飞书群

这意味着系统不仅记得你之前的操作（跨会话记忆），还能主动从团队的知识库中获取上下文。当 `technical_writer` 在编制监测日报时，它可以参考飞书知识库里的"监测日报模板"和"甲方格式要求"；当 `solution_architect` 在编写监测方案时，它能检索到团队之前类似项目的方案文档。

**企业知识 + 项目记忆 + 专业工具 = 一个真正了解你们团队工作方式的工程 AI。**

## 四、实战场景：一份监测日报的诞生

下面用一个完整的监测日报生成流程，展示各智能体是如何接力协作的。

**用户输入：**
> "Chief，这是本周地铁3号线基坑的自动全站仪监测数据 `week12_robot.csv`，请出一份本周的监测日报。"

**Step 1：Chief 接收并拆解任务**

`chief_manager` 输出执行计划：
```
【任务拆解与执行计划】
1. 调用 qa_inspector → 外业数据首检
2. 调用 data_analyst → 数据平差计算与趋势分析
3. 调用 technical_writer → 编制监测日报
4. 调用 qa_reviewer → 规范合规性终审
5. 汇总交付
```

**Step 2：外业质检员首检**

`qa_inspector` 检查 CSV 文件的数据完整性：测站信息是否齐全、是否存在大面积丢包、是否有仪器断电导致的基准漂移。如发现连续缺失超过 2 小时的数据段，将标记为无效区间。

**Step 3：数据分析师处理**

`data_analyst` 接收已通过首检的数据后：
- 调用 `monitoring_csv` 工具处理 CSV 海量数据（避免直接让 LLM 逐行读取几万行数据）
- 调用 `deformation_rate` 工具计算各测点的变形速率和趋势拟合
- 调用 `inclinometer` / `axial_force` / `water_level` 等工具分别处理测斜、轴力、水位等专项数据
- 调用 `survey_calculator` 的 `alert_level` 工具判定预警等级
- 调用 `chart_generator` 生成 SVG 趋势折线图
- 调用 `excel_export` 将数据汇总表导出为标准 .xlsx 文件（支持多 Sheet、冻结表头、自动筛选）

**Step 4：技术文档员成文**

`technical_writer` 将计算结果整合为标准格式的监测日报，包含：报告头部（项目名称、编号、日期、编制/审核/批准栏）、数据表格（测点编号、本期变化量、累计变化量、速率、预警等级）、趋势研判结论、下步监测建议。所有超标数据加粗标注并配 ⚠️ 标记。最终报告可通过 `report_export` 导出为 DOCX 文件，监测数据附表通过 `excel_export` 同步导出为 Excel。

**Step 5：总工终审**

`qa_reviewer` 逐项审查：报警值引用是否与方案一致、术语是否规范（"沉降量"与"沉降值"不能混用）、单位是否全文统一、预警等级判定是否准确。审查通过后，`chief_manager` 汇总输出最终交付物。

【配图6：监测日报生成的完整流程图——以时间线形式展示从用户输入→chief拆解→qa_inspector首检→data_analyst计算（调用monitoring_csv/deformation_rate/inclinometer/axial_force/water_level/chart_generator/excel_export等工具）→technical_writer成文（report_export导出DOCX + excel_export导出数据附表）→qa_reviewer终审→最终交付的全过程。每个节点标注对应的智能体名称和关键动作。】

## 五、技术选型

**底座框架**：RAILWISE-CLI 基于 [opencode](https://github.com/sst/opencode)（MIT 协议）深度定制。opencode 是一个成熟的 AI CLI 框架，提供了智能体定义、工具注册、会话管理等基础设施。我们在此基础上构建了面向测绘行业的完整上层应用。

**运行环境**：使用 [Bun](https://bun.sh) 作为 JavaScript 运行时，替代 Node.js。Bun 的启动速度和文件 I/O 性能显著优于 Node.js，在处理大体量监测数据文件时优势明显。

**模型支持**：系统兼容多家大模型提供商。特别值得一提的是，我们接入了四家国产免费模型——智谱 GLM（永久免费）、DeepSeek（注册送 500 万 tokens）、MiniMax 和 Kimi。这意味着团队成员无需任何 API 费用即可开始使用系统。当然，如果对生成质量有更高要求，也可以切换到 Claude、GPT-4o 或 Gemini。

**插件化架构**：所有智能体定义（Markdown + YAML frontmatter）和计算工具（TypeScript 文件）都放在项目根目录的 `.railwise/` 文件夹下，支持热加载。添加一个新的计算工具只需要在 `.railwise/tool/` 下新建一个 `.ts` 文件，导出 `tool()` 函数即可，无需修改任何核心代码。

**项目已在 GitHub 开源**：[github.com/railwise-cn/RAILWISE-CLI](https://github.com/railwise-cn/RAILWISE-CLI)

## 六、安装与使用

RAILWISE-CLI 已发布至 npm，支持 6 个平台的预编译二进制文件（linux-x64、linux-x64-baseline、linux-arm64、darwin-arm64、darwin-x64、windows-x64），一条命令即可安装：

**npm 安装（推荐）**

```bash
npm install -g railwise-ai
```

安装完成后，在任意目录运行 `railwise` 即可启动。系统会自动下载与你操作系统匹配的预编译二进制文件，无需手动编译。

**其他安装方式**

```bash
# curl 一键安装（Linux / macOS）
curl -fsSL https://raw.githubusercontent.com/railwise-cn/RAILWISE-CLI/main/install.sh | sh

# Homebrew（macOS / Linux）
brew install railwise-cn/tap/railwise
```

首次运行时，系统会引导你配置 API 密钥。支持智谱 GLM（永久免费）、DeepSeek、MiniMax、Kimi 等国产模型，也支持 Claude、GPT-4o、Gemini 等国际模型。详细的模型配置方法见项目 README。
